/shape-product による製品判断。53件の UI 到達性ギャップの根は
「web はビューアか NLE か」の未決。全部埋めると CapCut の劣化コピー、
埋めないと「音声は?文字は?」が再発する。

## テーゼ

web UI は**知覚チューニングに完全、構造編集に意図的に不完全**。

- 感覚(UI が完全であるべき): 再生/字幕の見た目と位置/音量・ダッキング/
  クロップ枠/オーバーレイ位置/採否判断
- 構造(会話に残す): カット構成、ハイライト選定、B-roll 選択、書き出し、
  公開パック。構造系に必要なのは**操作ではなく結果の可視化**

対抗馬は CapCut。機能数で勝負せず「任せられて、確かめられる」で勝つ。

## 帰結

- 波2 = 感覚の完全化(音声パネル・字幕入口・color 近似・位置系)
- 波3 = 書き出し UI を**作らない**。代わりに (a) 書き出し結果カード
  (警告込み)を確認タブに残す最小 daemon ルート、(b) 行き止まりに
  「Claude に頼む」チップ(精密な依頼文をワンクリックでクリップボードへ)
- 作らないもの: razor/挿入編集、compose 構築 UI、書き出しウィザード

## 合否と反転

- 第一マイルストーン: 検証シナリオ1〜2で UI⇔会話の往復回数と
  行き止まり回数を計測。行き止まりゼロが合格
- 方向終了シグナル: 感覚系すら会話の方が速い場面が過半
- フレーム無効シグナル: ユーザーが構造編集を UI で直接いじりたいと
  繰り返し感じる → フル NLE 路線を正面から再検討
