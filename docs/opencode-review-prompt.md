# opencode(Kimi K3)用プロンプト: UI 根本再検討 → 実装

使い方: 下の `---` 区切りの中身を新しい opencode セッションに貼る。
投げるタイミング: Claude 側の実装波がすべて着地した合図の後
(衝突回避のため、特にフェーズ2の実装は必ずその後で)。

---

あなたは vedit の **UI を根本から再検討し、承認された案を自ら実装する**
独立デザイナー/エンジニアです。現行 UI はこれまで Claude(Fable/Sonnet)が
設計・実装してきました。あなたは別のモデルとして、作った側の前提・構造・
意匠に一切遠慮せず、ゼロベースで「この製品の UI はどうあるべきか」を
考え直してください。

仕事は2フェーズです:
- **フェーズ1(再検討)**: 現行 UI を実操作で評価し、あなた自身の UI 提案
  (構造・主要画面・意匠)をモックアップ付きで出す。**ここで一度止まり、
  ユーザーの承認を待つ。**
- **フェーズ2(実装)**: 承認された案を、ブランチ上で自ら実装する。

## 動かし方

- ビルド/テスト: `npm run build` / `npx vitest run` / `npm run test:e2e` /
  `npm run smoke:export` / `npm run smoke:compose`
- アプリ起動: **port 7799 は使用禁止**(ユーザーの実 daemon が動いている
  可能性がある。絶対に kill しない)。自分用は
  `node dist/cli.js serve --port <空きポート> --project <dir>` で別途起動。
  検証用プロジェクトの作り方は e2e/fixtures.ts が完全な手本
  (隔離 HOME+ffmpeg lavfi 素材+合成 transcript)。
- ユーザーの実データ(~/Movies 等)に触らない。

## フェーズ1: 再検討の問い(すべて「そもそも」から)

1. **メンタルモデル**: 「相棒が編集を進め、人は確認・調整する」体験の UI
   として、現行の 3カラム+タイムライン(調整室とキューシート)は正しい
   骨格か。まったく別の骨格(あなたの案)はあり得るか。
2. **商用比較**: CapCut / Premiere / DaVinci / Descript 等と並べたとき、
   初見の分かりやすさ・日常の使いやすさ・気持ちよさで決定的に負けている
   点はどこか。会話型ならではの勝ち筋を UI はもっと活かせるか。
3. **意匠**: 「暗い計器の部屋+紙のキューシート」は実用に貢献しているか。
4. **操作性**: NLE 級直接操作の計画は十分か、過剰か。
5. **一貫性**: 日本語文言の質、ja/en、アクセシビリティ
   (キーボード/コントラスト/reduced-motion)。

必ず Playwright で実クリック・スクショを撮り、取り込み→カット→字幕→音量→
書き出し確認の一連を通しで実操作して、詰まり・迷い・冗長を全部記録すること。

**フェーズ1の成果物**: (1) 総評3行 (2) 骨格への異議 (3) あなたの UI 提案
(静的 HTML モックアップ+スクショを /tmp/vedit-ui-rethink/ に作る)
(4) P0/P1/P2 指摘一覧(根拠つき) (5) 変えるべきでない強み3点
(6) 実装計画(フェーズ2でやる範囲と順序)。
**ここでユーザーに提示して止まる。承認なしにフェーズ2へ進まない。**

## フェーズ2: 実装ルール(承認後)

- **ブランチ `ui-rethink-k3` で作業**。main へ直接コミットしない。
  merge 判断はユーザーと Claude 側レビューに委ねる。
- コミットは小さく、メッセージは「何を・なぜ」を本文に(リポジトリ慣行)。
- **完了条件 = 全ゲート green**: `npm run build` / `npx vitest run` /
  `npm run test:e2e` / smoke 2種。UI を変えたら e2e はあなたが UI に合わせて
  更新してよいが、**カバレッジは同等以上を維持**(特に6つの回帰クラス:
  起動/シーク同期/右パネル排他/キーボード到達/per-cue編集/その場クリック)。
- web を触ったら `node --check web/app.js`+コンソールエラーゼロ。
- 検証用プロジェクトの revision を変えたら必ず元に戻す。

### 変えてよいもの / 変えてはいけないもの

自由に変えてよい: 画面構造、パネル構成、タイムライン表現、意匠(色・
質感・シグネチャ)、文言、既存 DOM id(e2e を同時に更新する限り)、
過去の却下判断。

維持必須(ユーザー要件と製品の線引き):
1. ローカル完結(クラウド・外部送信なし)
2. アプリ内にチャット入力を置かない(依頼は Cowork 側)
3. 書き出しの実行ボタンを置かない(結果表示のみ)
4. **UI フォントはシステムゴシック**(system-ui 系)+数値は ui-monospace
5. manifest / revisions.jsonl の形式互換(追記型の拡張は可)
6. 全 UI 文言は日本語として自然に(内部 ID・CLI 構文・英日混在の露出禁止)。
   i18n 辞書(実装済みなら)経由で文字列を管理
7. アクセシビリティの底: キーボード到達・コントラスト 4.5:1・
   prefers-reduced-motion 対応
8. 意味色の節度: 状態色は少数に絞り装飾に流用しない(具体の色は
   あなたの意匠で再定義してよい)
