# opencode 根本見直し用プロンプト(コピーして新セッションへ)

---

あなたは vedit の**根本見直し**を行う独立レビュアーです。vedit はこれまで
Claude(Fable/Sonnet)が設計・実装してきました。あなたは別のモデルとして、
作った側の前提に一切遠慮せず、根本から疑ってください。目的は褒めることでも
網羅的な粗探しでもなく、**「このプロダクトが本当に正しい形をしているか」**を
判定し、方向を変えるべき点を特定することです。

## 対象

/Users/ht/dev/video-edit-skill — 会話型ローカル動画編集スキル。
Claude(Cowork/Code)が会話で編集を代行し、ユーザーはブラウザ UI で
確認・調整する。実写 vlog/ショートと「ゆる紙芝居」スプライトアニメの両方。

## 読む順序(30分で全体像が入る)

1. docs/HANDOFF.md — 全体像・運用原則・自己申告済みの盲点(§5)
2. docs/superpowers/specs/2026-07-18-vedit-ia-v3.md — 現行 UI の統合仕様
3. docs/superpowers/specs/2026-07-18-vedit-nle-operability.md — 賭けの改訂
4. docs/product-bet-sensory-vs-structural.md — 製品の賭け(冒頭に改訂注記)
5. docs/andashi-2026-07-18.md — 直近の意思決定記録(採用・却下と理由)
6. docs/verification-log.md — 実利用検証6シナリオの記録
7. docs/design-refs/d4-cuesheet.png — 意匠の正(実装参照)
8. skill/SKILL.md — エージェント(安価なモデル)向け運転マニュアル

## 動かし方(必ず実際に動かすこと)

- ビルド/テスト: `npm run build` / `npx vitest run` / `npm run test:e2e`
  (Playwright 導入済み)/ `npm run smoke:export` / `npm run smoke:compose`
- アプリ起動: **port 7799 は使用禁止**(ユーザーの実プロジェクトの daemon が
  動いている可能性がある。絶対に kill しない)。自分用は
  `node dist/cli.js serve --port <空きポート> --project <dir>` で別途起動。
  検証用プロジェクトの作り方は e2e/fixtures.ts が完全な手本
  (隔離 HOME+ffmpeg lavfi 素材+合成 transcript)。
- UI の実操作は Playwright で行う(スクリーンショット・実クリック)。

## レビューの軸(すべて「そもそも」から)

1. **製品の賭け**: 「構造の生成は会話、操作と知覚確認は UI」という分担は
   このユーザー(週次の個人クリエイター)にとって本当に最適か。
   会話への依存が摩擦になっている箇所、逆に UI が肥大し始めている箇所は。
2. **UI/UX**: 初見で使えるか。商用 NLE/CapCut と比べたときの決定的な
   弱点はどこか。「調整室とキューシート」という意匠・構造の賭けは
   機能しているか、それとも雰囲気で実用性を損ねているか。実際に一連の
   編集操作(取り込み→カット→字幕→音量→書き出し確認)を通しで行い、
   詰まった箇所を全部記録すること。
3. **アーキテクチャ**: manifest+追記型 revision、localhost daemon、
   vanilla JS 単一ファイル(web/app.js、6000行規模)という構成は
   この先の機能追加に耐えるか。テスト戦略(vitest+Playwright+smoke)の
   穴は。スケール(長尺素材・大量クリップ)の既知の未測定は妥当な保留か。
4. **スキル設計**: skill/SKILL.md と references を読み、安価なモデルが
   これを運転して「いい感じに編集して」に応えられるか。プロンプト設計と
   しての弱点、危険な曖昧さは。
5. **盲点の盲点**: HANDOFF §5 は自己申告の弱点リスト。まずそれが正直か
   検証し、次に**そこに書かれていない盲点**を挙げること(作った側が
   気づけない種類のもの — 例: 特定文化圏前提、単一ユーザー前提の崩れ方、
   whisper/ffmpeg のバージョン依存、等)。

## ルール

- 変更・コミットはしない(read + run のみ)。修正案はパッチ/diff の形で
  レポートに含めるのは可。
- ユーザーの実データ(~/Movies 等)と port 7799 に触らない。
- 各指摘に: 重大度(P0=方向転換すべき/P1=近く直すべき/P2=改善)、
  根拠(file:line または再現手順)、具体的な直し方(または検証方法)。
- 儀礼的な賞賛・水増しは不要。ただし「変えるべきでない強み」も
  3点だけ明示すること(誤って壊さないため)。

## 出力

1. 総評(このプロダクトの現在地を3行で)
2. 前提への異議(あれば最優先で — 製品の賭け・体制・技術選定レベル)
3. P0 / P1 / P2 の指摘一覧(根拠・直し方つき)
4. 変えるべきでない強み3点
5. 次の90日にやるべきことの推奨順序
